02 — Key Issues

Identified Usability Issues

Issues identified through both heuristic evaluation and usability testing, sorted by severity.

🟠 Major Issues (Severity 3)

No Cancel or Undo Option in Schedule
Severity 3
On the "My Schedule" screen, users have no way to go back a step or cancel an action. There is no Cancel button or Redo option, meaning any accidental action on the schedule cannot be reversed without starting over.
🔗 Related Heuristic: H3 — User Control and Freedom
No Input Validation on Sign Up Form
Severity 3
The Sign Up form asks for "Birth of date" and "Phone Number" without any input masks or format examples. Users can submit incorrect formats with no real-time constraints or guidance shown before clicking Register.
🔗 Related Heuristic: H5 — Error Prevention
Missing Empty States and No Search Error Feedback
Severity 3
When viewing "My Schedule" on a day with no trips, the screen appears broken rather than showing a helpful empty state message. Similarly, when a map search fails to find an attraction, the map simply remains static with no error or feedback message shown to the user.
🔗 Related Heuristic: H9 — Help Users Recognize, Diagnose, and Recover from Errors

🟡 Minor Issues (Severity 2)

Incorrect Active Navigation State on Home Screen
Severity 2
On the Home screen, the bottom navigation bar highlights the "Search" icon (magnifying glass) as active, but the displayed page content shows the "Explore" dashboard — not a search interface. This misleads users about where they currently are in the app.
🔗 Related Heuristic: H1 — Visibility of System Status
Ambiguous "Done" Button on Profile Edit Screen
Severity 2
On the Profile edit screen, both a "Done" button and a "Back" arrow are present at the top. It is unclear whether "Done" saves the changes or if tapping the "Back" arrow discards them, causing confusion about the outcome of each action.
🔗 Related Heuristic: H2 — Match Between System and the Real World
Inconsistent Naming of Destination Labels
Severity 2
One screen uses the label "Popular Places" while the home dashboard refers to the same content as "Best Destination." This inconsistency in terminology makes the interface feel unprofessional and may confuse users about whether these are different categories.
🔗 Related Heuristic: H4 — Consistency and Standards
Search Bar Cleared After Map Navigation
Severity 2
On the Map Search screen, once a user clicks into the map (MAP1 view), the search bar resets to empty. Users who navigate through multiple map layers may forget what they originally searched for, increasing memory load unnecessarily.
🔗 Related Heuristic: H6 — Recognition Rather Than Recall
Visual Overload in Place Details and Schedule Screen
Severity 2
The "Luksohon Falls" detail sheet contains redundant action buttons — both "Directions" and "Start" — cluttering the primary view. On the "My Schedule" screen, large date text placed next to tiny location addresses creates a disjointed and hard-to-read layout.
🔗 Related Heuristic: H8 — Aesthetic and Minimalist Design
No Tooltips or Onboarding Prompts for Map Features
Severity 2
There are no tooltips or onboarding prompts to help first-time users understand how to use the interactive map or the schedule sync feature. New users are left to discover these functions on their own, with no contextual guidance available.
🔗 Related Heuristic: H10 — Help and Documentation

⚪ Cosmetic Issues (Severity 1)

No Shortcut Filters for Common Tourist Needs on Map
Severity 1
The Map view includes zoom buttons and a re-center icon, but there are no accelerators or shortcut "chips" for frequent tourist needs such as Restrooms, Food, or Parking. Expert users have no fast-access filters directly on the map interface.
🔗 Related Heuristic: H7 — Flexibility and Efficiency of Use
03 — Discussion

Discussion of Findings

Analysis and interpretation of the evaluation results in the context of HCI principles.

Overall Usability Assessment

PartidoGO received an average System Usability Scale (SUS) score of 62.1 out of 100 from 20 respondents, placing it in the Below Average range based on the standard SUS interpretation scale. While the prototype was generally understood in terms of its purpose as a tourism guide for the Partido area in Camarines Sur, the evaluation revealed recurring usability gaps that affected the overall user experience. Notably, 2 respondents from the BSIT course awarded perfect scores of 100, while several BET-EET respondents rated the app below 52.5, indicating a significant gap in perceived usability between user groups with varying technology familiarity.

The heuristic evaluation conducted by all four group members identified 10 distinct usability violations across the prototype, covering all of Nielsen's 10 heuristics. The most frequently violated heuristics were H1 (Visibility of System Status), H3 (User Control and Freedom), and H5 (Error Prevention), each receiving a severity rating of 3 — classified as major usability problems that require priority attention. These findings are consistent with the SUS results, where users expressed difficulty with navigation flow, unclear feedback, and the absence of error recovery mechanisms while browsing tourist spots and managing their itineraries.

Overall, PartidoGO shows strong potential as a local tourism companion for the Partido region of Camarines Sur. The app's core concept is clear and its visual design was positively received by several respondents. However, the current prototype requires targeted improvements — particularly in navigation consistency, input validation, and user control — before it can be considered fully usable by its intended target audience of tourists and first-time visitors.

Relationship to HCI Principles

The evaluation findings directly reflect several of Jakob Nielsen's 10 Usability Heuristics, confirming that the identified issues are rooted in established HCI principles rather than isolated user preferences. The most critical violation found was under H1 — Visibility of System Status, where the Home screen's bottom navigation bar highlighted the "Search" icon as active while the Explore dashboard was displayed — a direct failure to keep the user informed of the system's current state. This type of mismatch creates confusion and erodes user trust in the interface.

H3 — User Control and Freedom was violated in the My Schedule screen, where users had no Cancel button or undo option when creating or editing an itinerary. According to Nielsen, users need clearly marked "emergency exits" to leave unwanted states — their absence in this screen forces users into a one-way flow with no recovery path. Similarly, H5 — Error Prevention was violated in the Sign Up screen, where the absence of input masks for date and phone number fields allowed users to submit incorrectly formatted data without any prior warning, directly contradicting the principle that the best designs prevent errors before they occur.

H4 — Consistency and Standards was flagged by multiple evaluators due to the inconsistent use of "Popular Places" on one screen and "Best Destination" on another to describe the same content. This violates the expectation that users should not have to wonder whether different words mean the same thing. H6 — Recognition Rather Than Recall was also noted in the Map Search screen, where the search bar reset to empty after navigating between map layers, unnecessarily increasing the user's cognitive load by requiring them to remember and re-enter their previous query.

Collectively, these findings indicate that the primary usability challenges in PartidoGO stem from insufficient system feedback, limited user recovery options, and inconsistent interface language — all of which are well-documented usability concerns in HCI literature. Addressing these heuristic violations through the recommended design changes would meaningfully raise the app's SUS score and make PartidoGO a significantly more intuitive tourism companion for the Partido region.

Evaluation Process

Step 1
Prototype Preparation
The PartidoGO mobile app prototype was prepared using Figma prior to the evaluation session. The prototype covers core features including browsing tourist spots around the Partido area, viewing location details and photos, and navigating between different destination categories in Camarines Sur. The prototype was reviewed by all group members before the session to ensure it was ready for presentation.
Step 2
Heuristic Evaluation
Each of the four group members independently reviewed the PartidoGO prototype using Nielsen's 10 Usability Heuristics. Violations were documented with the specific screen name, a description of the issue, and a severity rating from 0 (not a problem) to 4 (usability catastrophe).
Step 3
Participant Recruitment
Twenty respondents were recruited, primarily from the EET (Electronics Engineering Technology) course, with additional participants from BSIT and BSOA programs for the usability testing session. These participants were selected as they represent potential users of the app — individuals with basic to intermediate smartphone experience who could provide relevant and reliable feedback on the PartidoGO prototype's usability. None of the participants had prior exposure to the prototype before the session.
Step 4
Usability Testing Sessions
The usability testing was conducted through Google Meet. The group leader presented and walked through the PartidoGO prototype live on screen, demonstrating its core features including browsing tourist spots, viewing destination details, and navigating across different categories within the Partido area. The participants observed and listened as the prototype was presented to them. After the session, a Google Form questionnaire was sent to all participants to gather their feedback and usability ratings.
Step 5
Data Analysis
The responses collected from the Google Form questionnaire were compiled and analyzed. SUS scores were computed per participant and averaged to determine the overall usability rating of the PartidoGO prototype. The results were then compared with the findings from the heuristic evaluation to identify the most critical design issues requiring improvement.

04 — Recommendations

Design Recommendations

Proposed improvements to address the usability issues identified during evaluation.

🔴 High Priority (Address Immediately)

🗺️

Fix Active Navigation Icon to Match Current Screen

The bottom navigation bar currently highlights the "Search" icon as active even when the "Explore" dashboard is displayed. Update the navigation logic so the active icon always reflects the screen the user is currently viewing. (H1 — Visibility of System Status)

✅

Add Cancel Button and Reset Option in Schedule Screen

Users currently have no way to exit or undo changes while creating or editing an itinerary in the My Schedule screen. Add a "Cancel" or "Close (X)" button to allow users to exit without saving, and include a "Reset Schedule" button to let them start over without deleting entries one by one. (H3 — User Control and Freedom)

🔒

Add Input Constraints to Sign Up Form

The Sign Up form asks for "Birth Date" and "Phone Number" with no format guidance or input masks. Add input masks (e.g., mm/dd/yyyy for date, +63-XXX-XXXX for phone) and inline validation messages to prevent users from submitting incorrect data before clicking Register. (H5 — Error Prevention)

🟡 Medium Priority

📋

Standardize Terminology Across All Screens

The app inconsistently uses "Popular Places" on one screen and "Best Destination" on the home dashboard to refer to the same content. Choose one term and apply it consistently across all frames, buttons, and labels to avoid confusing users. (H4 — Consistency and Standards)

💾

Rename "Done" Button to "Save Changes" in Profile Screen

The "Done" button on the Profile edit screen does not clearly communicate that it saves the user's changes. Rename it to "Save Changes" to make the action explicit, and clarify whether the "Back" arrow discards or keeps unsaved edits. (H2 — Match Between System and the Real World)

🔍

Persist Last Search Term in Map Search Bar

When a user searches for an attraction and then navigates through the map layers, the search bar resets to empty. Keep the last searched term visible in the search bar, or add a "Recent Searches" dropdown so users do not need to re-enter their query after moving across map views. (H6 — Recognition Rather Than Recall)

🟢 Low Priority / Future Enhancements

🏕️

Add Quick Filter Chips on the Map for Tourist Needs

The map currently only has zoom and re-center controls but no shortcut filters for common tourist needs. Add small filter chips or icons directly on the map interface for categories like Restrooms, Food, and Parking so users can quickly find what they need without leaving the map view. (H7 — Flexibility and Efficiency of Use)

🧹

Simplify Place Details Sheet and Clean Up Schedule Layout

The "Luksohon Falls" detail sheet shows both a "Directions" and a "Start" button which serve the same purpose. Combine them into one primary action button. On the My Schedule screen, improve text hierarchy by balancing the size of dates and location addresses for a cleaner, more readable layout. (H8 — Aesthetic and Minimalist Design)

💬

Add Empty State Messages and Onboarding Tooltips

When a user opens My Schedule with no trips, the screen looks broken rather than intentionally empty. Add a friendly empty state message like "No trips yet! Explore destinations to add them here." Also add a one-time onboarding tooltip on the Map screen to guide first-time users on how to use the interactive map and schedule sync. (H9 & H10 — Error Recovery and Help & Documentation)
