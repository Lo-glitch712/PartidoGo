Usability Testing — PartidoGO
Partido State University · CEC · AY 2025-2026

Study Title

PartidoGo is a mobile application designed to streamline tourism in the Partido area of Bicol by allowing users to easily locate and book local resorts and tourist spots. It solves the difficulty of finding reliable travel information in the region, specifically targeting local travelers and tourists looking for a seamless exploration experience. Core features include an interactive map, verified accommodation listings, and a simplified booking system.

Purpose of the Study

This study aims to evaluate the usability of the PartidoGO mobile application prototype. The evaluation will help identify usability issues and areas for improvement based on user interaction and feedback.

Confidentiality

All information collected during this study will be kept confidential and will only be used for academic purposes. Your name will not be associated with any published results unless you give explicit permission.

Voluntary Participation

03 — Instruments

Usability Testing Setup

The usability testing was conducted remotely via Google Meet. After each session, participants answered the System Usability Scale (SUS) through a Google Form.

📹

Google Meet

Sessions were conducted online via Google Meet. Participants shared their screens while navigating the PartidoGO Figma prototype. Evaluators observed in real time using a think-aloud protocol.

Remote Testing Screen Sharing Think-Aloud Protocol Live Observation
📋

Google Forms — SUS Survey

After each session, participants answered the System Usability Scale (SUS) — a 10-item questionnaire rated on a 5-point Likert scale — plus two open-ended questions, all via Google Forms.

🗓️ Session Process

1
Invitation & Consent
Participants were invited via Google Meet. The consent form was explained and acknowledged before the session began.
2
Prototype Testing via Screen Share
Participants shared their screen and navigated the PartidoGO Figma prototype while thinking aloud. Evaluators observed silently and noted issues.
3
Post-Test SUS Survey via Google Forms
After the session, participants were sent the Google Form link containing the SUS questionnaire and open-ended questions.

# Heuristic Issue Found Screen Severity Recommendation
H1 Visibility of System Status The bottom navigation bar highlights the "Search" icon (magnifying glass) as active, but the page content shows the "Explore" dashboard rather than a search interface. Home Screen 3 Ensure the active state of the navigation icons correctly matches the current page content.
H2 Match Between System and the Real World The "Done" button and "Back" arrow are both present at the top, but it isn't immediately clear if "Done" saves the changes or if the "Back" arrow discards them. Profile 2 Use more explicit labels like "Save Changes" instead of "Done" to ensure the action matches the user's intent to update their information.
H3 User Control and Freedom The user has no way to go back a step in the process. There is no cancel button or option to redo the schedule. My Schedule 3 Include a "Cancel" button or a "Close" (X) icon when the user is creating or editing an itinerary. Add a "Reset Schedule" button or "Edit" mode to modify individual entries without deleting the entire plan.
H4 Consistency and Standards Naming inconsistency: the header on one screen says "Popular Places" while the home dashboard refers to them as "Best Destination". Multiple Screens 2 Standardize button components and terminology (e.g., choose either "Places" or "Destinations") across all frames to maintain a professional look and feel.
H5 Error Prevention The "Sign up" form asks for "Birth of date" and "Phone Number" immediately. If a user enters an invalid format, there are no visible constraints or examples to guide them before they click Register. Sign Up 3 Use input masks for the date (mm/dd/yyyy) and phone number fields to prevent users from submitting incorrect data.
H6 Recognition Rather Than Recall In the map, the search bar contains the placeholder "Search attractions...". Once the user clicks into the map, the search bar is still visible but empty. Users might forget what they were originally looking for if they navigate through multiple map layers. Map Search 2 Keep the last searched term visible in the search bar or provide a "Recent Searches" dropdown to minimize the user's memory load.
H7 Flexibility and Efficiency of Use The MAP view shows zoom buttons (+ and -) and a "re-center" icon, but there are no accelerators for power users such as category filters (e.g., "Nearby Food" or "Gas Stations") directly on the map. Map 2 Add small shortcut "chips" or icons for common tourist needs (Restrooms, Food, Parking) directly on the map interface to speed up interaction.
H8 Aesthetic and Minimalist Design The "Luksohon Falls" sheet contains redundant action buttons (both a "Directions" button and a "Start" button) that clutter the primary view. On "My Schedule", the use of large dates next to tiny location addresses creates a disjointed reading flow. Map · Schedule 2 Combine "Directions" and "Start" into a single primary action button. Reduce the number of icons and secondary text details displayed on initial cards to make the interface feel more modern and less busy.
H9 Help Users Recognize, Diagnose, and Recover from Errors If a user views "My Schedule" on a day with no trips, the screen might appear broken rather than simply empty. There is also no clear feedback if a search for an attraction fails — the map simply remains static. My Schedule · Map 3 Display a friendly empty state message like "No trips today! Explore new destinations." Use traditional error styling (red text or icon) if a user attempts to "Set Schedule" without selecting a valid date.
H10 Help and Documentation There are no tooltips or onboarding prompts to explain how to use the interactive map or the schedule sync feature. Map · Schedule 2 When a user first opens the Map, provide a one-time "coach mark" or tooltip pointing to the "View Maps" button to explain its function right at the moment they need it.
Severity Scale: 0 — Not a problem 1 — Cosmetic only 2 — Minor 3 — Major 4 — Catastrophe

# Heuristic Issue Found Screen Severity Recommendation
H1 Visibility of System Status The app shows navigation labels (Maps, Home, Calendar, Profile) but lacks detailed instructions or guides. The scheduling feature provides confirmation messages, but other parts of the app still need clearer and more consistent feedback after user actions. Multiple 2 Add clearer feedback and simple guides for first-time users. Include loading indicators and confirmation messages so users know what is happening in the system.
H2 Match Between System and the Real World The prototype uses simple words, familiar icons, and images that help users understand the app naturally. The design follows real-world conventions and users can navigate easily without confusion. All Screens 1 Continue using simple and familiar words, icons, and layouts. Organizing information more clearly would also make navigation easier for users.
H3 User Control and Freedom Most features lack user control because there are only limited exit, reset, undo, and redo options. This is especially important in booking hotels, restaurants, and resorts where users may need to edit or cancel their decisions. Schedule · Booking 3 Add more options such as cancel, back, or edit buttons so users can easily correct mistakes or change decisions, especially when scheduling or booking trips.
H4 Consistency and Standards Some parts of the app use different visual styles and layouts, especially in the bottom navigation and interface sections. These inconsistencies may confuse users and affect the overall user experience. Navigation 2 Use the same colors, layouts, and button styles consistently throughout the app to avoid confusing users and make the interface look more organized.
H5 Error Prevention The app only provides confirmation and conditions during login and signup. Other features, especially scheduling and booking, still lack proper error prevention tools to help users avoid accidental mistakes. Schedule · Booking 3 Include more confirmation prompts and warnings before users complete important actions to help prevent accidental mistakes or incomplete inputs.
H6 Recognition Rather Than Recall Some parts of the app use similar layouts and repeated feature designs which help users become familiar with the interface. Commonly used functions are visible and recognizable so users do not need to memorize too many steps. All Screens 1 Keep important buttons and features visible at all times to help users navigate the app more easily without needing to remember too many steps.
H7 Flexibility and Efficiency of Use The app is simple and accessible but the efficiency of the experience depends on how users interact with it. Advanced shortcuts or personalized features are still limited. All Screens 2 Allow users to save favorites or quickly access frequently used features for a smoother and more efficient experience.
H8 Aesthetic and Minimalist Design The prototype focuses on necessary content and avoids unnecessary elements. However, some sections may still be improved to make the interface look cleaner and more visually balanced. Multiple 1 Improve the spacing, alignment, and overall arrangement of elements to make the interface look cleaner and more visually balanced while keeping its simple design.
H9 Help Users Recognize, Diagnose, and Recover from Errors The current prototype still lacks proper error messages and recovery options. These features are important because they help users understand problems and guide them on how to fix mistakes quickly. All Screens 3 Add clear error messages with simple explanations and solutions to help users understand problems and recover from mistakes more easily.
H10 Help and Documentation The prototype currently does not provide help or documentation features. These can still be added in the settings or support section to guide users whenever they need assistance while using the app. Settings 2 Include a simple help section or FAQ page so users can quickly find guidance whenever they need assistance while using the system.
Severity Scale: 0 — Not a problem 1 — Cosmetic only 2 — Minor 3 — Major 4 — Catastrophe
